# Crystalline Integration — Analyse

## Was ist Crystalline?

Crystalline ist ein LSP (Language Server Protocol) für Crystal, geschrieben in Crystal. Es nutzt den Crystal-Compiler direkt für Type-Inference und Diagnostics.

**Repo:** https://github.com/elbywan/crystalline (513 Stars, 35 Forks)
**Aktuelle Version:** 0.18.0 (Crystal 1.20 kompatibel)
**Status:** Passiv maintained — Autor hat klargestellt, dass er nicht mehr aktiv Features hinzufügt oder Bugs fixt

## Features von Crystalline

| Feature | Status | IntelliJ Plugin hat es bereits? |
|---------|--------|--------------------------------|
| **Code Diagnostics** (Syntax + Semantik) | ✅ Implementiert | ❌ Nein — das wäre **NEU** |
| **Autocompletion** (Methoden, Macros, Klassen) | ✅ Limited | ✅ Eigenständig, besser |
| **Formatting** | ✅ via `crystal tool format` | ✅ Bereits delegiert |
| **Go to Definition** | ✅ Implementiert | ✅ Eigenständig, mit StubIndex |
| **Hover Information** (Typen, Signaturen) | ✅ Implementiert | ❌ **NEU** — zeigt Typen beim Hovern |
| **Document Symbols** | ✅ Implementiert | ✅ Structure View |

## Kern-Features die Crystalline NEU bringen würde

### 1. Echte Code-Diagnostics (Hauptvorteil)
- **Ameba-Linting** — Crystalline integriert ameba implicit
- **Semantische Fehler** — echte Compile-Fehler als Inline-Markierungen
- **Typfehler** — z.B. `String` statt `Int32` übergeben
- **Vorteil:** Der aktuelle Plugin hat eine eigene einfache Typprüfung, aber Crystalline nutzt den echten Compiler → **100% korrekte Fehler**

### 2. Hover Information
- Maus über Variable → Typ-Anzeige
- Maus über Methode → Signatur + Dokumentation
- **Vorteil:** Sehr nützlich für Crystal wegen impliziten Typen

### 3. Ameba-Linting (Code Quality)
- Stilregeln (z.B. `Naming/TypeNames`, `Layout/TrailingBlankLines`)
- Code-Smells (z.B. `Lint/ShadowingOuterLocalVar`)
- **Vorteil:** Professionelle Code-Qualität direkt im Editor

## Limitierungen von Crystalline

| Problem | Auswirkung |
|---------|------------|
| **Hoher RAM-Verbrauch** | Crystal-Compiler + Boehm GC → 500MB-1GB pro Projekt |
| **Lange Compile-Zeiten** | Bei großen Projekten kann LSP 10-30s brauchen |
| **Nur aufgerufene Methoden** | Methoden die nirgends aufgerufen werden, werden nicht analysiert |
| **Kein inkrementelles Parsen** | Bei jedem Save wird Datei komplett neu analysiert |
| **Passiv maintained** | Keine Bugfixes, keine neuen Features zu erwarten |

## Integrations-Strategien

### Option A: Crystalline als externen LSP-Prozess starten
- Crystalline läuft als eigenständiger Prozess (`crystalline --stdio`)
- Plugin kommuniziert via LSP-Protocol (JSON-RPC)
- **Vorteil:** Saubere Trennung, Crystalline kann unabhängig upgedated werden
- **Nachteil:** Hoher Overhead (zwei Compiler-Instanzen), Crystalline's eigene Parsing-Fehler

### Option B: Ameba direkt integrieren (ohne Crystalline)
- `ameba` als externen Prozess aufrufen (like `crystal tool format`)
- Nur Linting-Ergebnisse als Inspections anzeigen
- **Vorteil:** Leichtgewichtig, kein LSP nötig
- **Nachteil:** Nur Linting, keine semantischen Diagnostics

### Option C: Eigener Fork von Crystalline
- Crystalline forken und für IntelliJ anpassen
- **Vorteil:** Volle Kontrolle
- **Nachteil:** Riesiger Aufwand, Crystalline ist in Crystal geschrieben (nicht in Java/Kotlin)

### Option D: Hybrid — Ameba + eigene Compiler-Diagnostics
- Ameba für Linting (externer Prozess)
- Crystal-Compiler für semantische Fehler (bereits teilweise implementiert via `crystal build --no-codegen`)
- **Vorteil:** Kein LSP nötig, eigene Kontrolle
- **Nachteil:** Compile-Zeiten, kein Hover

## Meine Empfehlung

**Option B oder D** sind am realistischsten:

1. **Ameba-Integration** als Inspection-Plugin (externer Prozess, schnelle Rückmeldung)
2. **Crystal-Compiler für semantische Diagnostics** (bereits teilweise vorhanden via `crystal build`)
3. **Hover Information** könnte man separat implementieren (PSI-basiert, ohne Crystalline)

Crystalline als LSP zu integrieren (Option A) wäre der "saubere" Weg, aber:
- Der Overhead ist enorm (Crystal-Compiler als externer Prozess)
- Crystalline ist passiv maintained
- Die community-Feedback ist gemischt
